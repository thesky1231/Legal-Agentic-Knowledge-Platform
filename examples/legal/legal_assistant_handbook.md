# 刑事法律知识助手示例手册

## 1. 抢劫与抢夺的区分

抢劫通常以暴力、胁迫或者其他足以压制被害人反抗的方式取得财物，核心在于行为对人身法益形成直接压制。  
抢夺则更强调乘人不备、公然夺取财物，一般不以持续压制被害人反抗为必要条件。  
在法律知识问答场景中，系统回答这类问题时应优先说明手段、对象和行为方式的差异，而不是只给出简短结论。

## 2. 非法拘禁与绑架的区分

非法拘禁重点在于非法剥夺他人人身自由。  
绑架通常还伴随以被害人为筹码，向家属、单位或者其他第三人提出财物、条件或者胁迫要求。  
如果案件材料里同时出现控制人身自由和索取财物、提出条件等情节，系统应提示用户结合完整事实进一步判断，而不是只凭单一片段直接定性。

## 3. 合同诈骗与证据不足时的保守回答

仅凭聊天记录、单方陈述、转账截图或者不完整合同文本，通常不足以直接认定合同诈骗。  
在法律助手场景中，系统遇到“能不能直接认定”“一定构成什么罪”“一定判几年”这类问题时，应优先采取保守回答策略，说明证据不足、需要补充案件事实和证据链。  
这类回答允许引用相关知识点和风险提示，但不应在证据薄弱时输出确定性结论。

## 4. 引用与回答规范

法律问答服务应尽量提供引用片段、出处章节和检索分数，方便用户回看依据。  
系统回答时优先走 grounded RAG 链路：先检索、再重排、再基于证据组织回答。  
如果 top hit 的证据分数不足，系统应明确提示“证据不足”或“无法直接得出确定性结论”，而不是补全想象内容。

## 5. 多 Agent 审核流程

在团队式 Agent 模式下，可以把链路拆成 `react-agent -> review-agent -> narration-agent`。  
`react-agent` 负责检索和起草回答；`review-agent` 重点检查 grounded 状态、citation 数量和回答是否过度推断；`narration-agent` 再把批准后的回答转成适合讲解的输出。  
这种方式尤其适合高风险问答场景，因为审核步骤是显式的、可记录的、可回放的。

## 6. 语音讲解与交付链路

当用户需要面向客户或培训对象讲解时，系统可以把 grounded answer 转成讲解脚本，再交给 TTS 和 A2F 渲染。  
这样同一条法律问答链路既能产出带引用的文字回答，也能产出适合语音播报或数字人演示的讲解结果。  
如果回答本身属于低置信或保守拒答，讲解链路也应保持同样的风险提示，不应把不确定内容包装成确定结论。
