# Interview Script ZH

## 一分钟版本

`我做的核心项目是一个面向法律知识问答场景的 Agentic Knowledge Platform。它不是简单的聊天机器人，而是一套后端服务：先把文档做结构化解析和知识入库，再通过 RAG 检索和重排生成带引用的回答，然后把回答接到单 Agent 或多 Agent 工作流里，同时做拒答控制、运行审计和指标观测。这个项目最重要的点不是功能多，而是我把它做成了一个更像真实交付系统的作品，而不是单纯 demo。`

## 三分钟版本

`我一开始有一个 Legal RAG 项目，重点在法律场景下的检索、引用和回答评测；后来我又做了一套更通用的 Agent 后端平台，里面有模型路由、单 Agent、多 Agent、运行审计和可观测性。最后我没有把它们当成两个分散项目，而是把它们合成了一个主项目，也就是现在的法律知识问答 Agent 平台。`

`这个项目的主链路是：文档解析 -> 切块 -> Embedding -> 向量检索 -> 重排 -> 带引用回答 -> reviewer 审核 -> narration 讲解。法律场景对幻觉和误答更敏感，所以我专门加了题型分类、拒答控制和低置信兜底，不会为了给答案而强行回答。`

`工程上我把它做成了可服务化的后端：有 FastAPI API、API Key 鉴权、运行审计、结构化日志、Prometheus 风格指标，还有模型路由层的 rate limiting、retry、circuit breaker 和 fallback。评测层我也合进来了，导入了原法律项目的回答集和检索黄金集。历史 benchmark 快照里，回答正确率、引用正确率和拒答适当率都是 100%，检索任务在 20 条样本上的 Recall@3/5 达到 95%。所以这个项目既能讲业务场景，也能讲工程化落地。`

## 八分钟展开版

### 1. 为什么选法律场景

`我刻意没有做泛知识问答，因为法律问答更能体现系统设计难点。它要求回答必须有依据，不能随意编造，也不能对证据不足的问题给出过度确定的结论，所以天然适合展示 citation grounding、拒答控制和评测。`

### 2. 整体架构怎么拆

`我把系统拆成了几层。最底层是文档解析、切块、Embedding 和向量存储；中间是知识库服务，负责检索、重排和生成 grounded answer；上面是模型路由层，负责针对不同任务做模型编排，并加入 rate limiting、retry、circuit breaker 和 fallback；再往上是 Agent 工作流，既支持单 Agent，也支持 react-agent、review-agent、narration-agent 的团队模式。`

### 3. 你重点解决了什么问题

`我重点解决的是三个问题。第一是幻觉控制，所以回答必须带引用，同时加了题型分类和低置信兜底。第二是服务稳定性，所以模型调用不是直接裸调，而是做了路由、限流、重试、熔断和 fallback。第三是可复盘性，所以我保留了运行审计、结构化日志和指标输出。`

### 4. 评测是怎么做的

`我不想让项目停留在“跑通了”，所以把原 Legal RAG 项目里的评测资产合并进来了。一部分是回答级评测，看回答是否正确、引用是否正确、是否出现幻觉、拒答是否合适；另一部分是检索级黄金集，看 Recall@3 和 Recall@5。这样我在面试里不只是说系统能回答，而是能说清楚这个系统怎么被验证。`

### 5. 为什么说它更像真实交付

`因为它已经具备了很多生产前必备的表达：有 API、鉴权、日志、指标、运行审计、多租户隔离、离线评测和多 Agent 协作。虽然它不是线上大规模业务系统，但它的设计方式是按可交付后端来做的。`

## 高频问题与回答

### 如果面试官问：为什么不直接做一个普通 RAG？

`因为普通 RAG 只能证明我会做检索问答，但不能证明我会做任务编排、稳定性控制和复杂工作流。这个项目里我故意把它往 Agent 后端方向推，就是为了展示从知识库到服务化再到执行工作流的完整能力。`

### 如果面试官问：为什么要做拒答？

`因为在法律场景里，错误的确定性回答比拒答更危险。如果证据不足或者问题本身要求系统做不该做的确定判断，我宁愿给保守回答，也不强行生成看起来完整但不可靠的答案。`

### 如果面试官问：你的评测数据从哪来？

`一部分来自我原 Legal RAG 项目积累的回答评测集和检索黄金集，后来我把它们迁移进现在这个主项目，用来验证系统在法律场景下的回答质量、引用质量和检索表现。`

### 如果面试官问：你觉得这个项目离真正上线还差什么？

`我会说它已经有比较完整的服务化骨架，但如果真上线，我会继续补异步任务队列、权限体系、真实 OCR/ASR 接入、数据库持久化和更完整的 tracing。重点不是说它已经完美上线，而是我知道它离生产还有哪些关键步骤。`

### 如果面试官问：这个项目里你最体现工程能力的部分是什么？

`我会答模型路由和评测体系。前者体现的是稳定性和服务治理能力，后者体现的是我不是只会搭链路，还会考虑怎么验证系统质量。`

## 面试里的表达原则

- 先讲业务风险，再讲技术设计
- 先讲主链路，再讲工程细节
- 先讲你解决了什么问题，再讲你用了什么框架
- 所有数字都说明来源，历史 benchmark 就讲历史 benchmark
- 不要把项目讲成“功能堆砌”
